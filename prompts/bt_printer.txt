Task:
Implement a Bluetooth thermal printer feature for `flutter-pos-demo` as an Android-first capability, with architecture ready to support restaurant mode and future business modes later.

Goal:
Add a production-grade printer foundation that supports:
- paired Bluetooth printer discovery
- connect / disconnect printer
- remember last selected printer
- test print
- print ESC/POS receipt bytes
- checkout success -> ask user whether to print receipt
- clear separation between receipt composition and printer transport

Important constraints:
- Keep Clean Architecture
- Keep Riverpod
- Keep go_router
- Do NOT put business logic in widgets
- Do NOT call Bluetooth plugin directly from checkout UI
- Printer integration must be abstracted behind repository/contracts
- Receipt composition must be separated from Bluetooth writing
- Android-first implementation is acceptable
- Design code so retail mode works first, but restaurant mode can reuse the same printer pipeline later

Dependencies to add:
- `print_bluetooth_thermal`
- `esc_pos_utils_plus`

High-level architecture:
1. `features/printer`
   - Bluetooth device discovery
   - connection lifecycle
   - persistence of selected printer
   - test print
2. `features/receipt`
   - compose ESC/POS bytes from transaction/receipt model
3. `features/checkout`
   - after successful checkout, show prompt to print
   - call printer flow via use case/service, not plugin directly

- get paired printers
- connect(printerId)
- disconnect()
- testPrint()
- printBytes(List<int> bytes)

3. Datasources
Create:
- Bluetooth datasource wrapping `print_bluetooth_thermal`
- Preferences datasource wrapping `shared_preferences`

Bluetooth datasource responsibilities:
- get paired devices
- connect/disconnect
- write bytes
- test basic Bluetooth state
- expose only clean app-facing models or raw plugin mapping internally

Preferences datasource responsibilities:
- persist default printer id/name
- load default printer info

4. Receipt composer

- `ReceiptComposer` abstract contract
- `RetailReceiptComposer` implementation using `esc_pos_utils_plus`

Retail receipt must support:
- store name
- transaction id
- date time
- line items
- subtotal
- VAT 7%
- grand total
- payment method
- cash received / change if available
- footer note

Do not hardcode printer calls inside composer.
Composer must only return ESC/POS bytes.

5. Receipt print service
Create a service/use case that:

8. Checkout integration
Modify checkout flow so that after successful payment:

- if user confirms:
  - print receipt
- if printing fails:
  - show user-friendly error/snackbar

Do not auto-print immediately in the first implementation unless already trivial and safe.
Prompt-first flow is preferred.
