Task:
Implement the **business mode architecture foundation** for `flutter-pos-demo` so the app can support multiple business modes in one codebase, starting with:
- `retail`
- `restaurant`

This task is about **mode architecture**, not full business feature implementation.

Main objective:
Create a clean, extensible mode-driven architecture that allows the app to:
- know which business mode is active
- expose feature capabilities by mode
- build navigation and routes by mode
- plug in pricing and receipt strategies by mode
- support mode-specific seed data later
- stay compatible with the existing retail-first demo flow

Important scope rules:
- Do NOT implement full restaurant features yet
- Do NOT build full table management, buffet flow, split bill, or kitchen ticket
- Do NOT move business logic into widgets
- Do NOT hardcode mode checks throughout the UI
- Do NOT over-engineer with unnecessary plugin systems
- Focus on a clean, practical foundation

High-level deliverables:
1. BusinessMode model
2. BusinessModeDefinition contract
3. FeatureCapability system
4. BusinessModeRegistry
5. Current active mode provider/state
6. Mode-aware routing foundation
7. Mode-aware navigation foundation
8. Pricing strategy abstraction hook
9. Receipt strategy abstraction hook
10. Seed data strategy abstraction hook
11. Minimal retail + restaurant mode definitions

Architecture constraints:
- Keep Clean Architecture
- Keep Riverpod
- Keep go_router
- Keep Isar
- Keep business logic out of widgets
- Keep current retail flow working
- Allow future modes such as cafe, clinic, warehouse without major refactor
- Use capability-based design instead of scattering `if (mode == ...)`

Detailed implementation requirements:

## 1. BusinessMode model

Create a base business mode identity.

Minimum:
```dart
enum BusinessMode {
  retail,
  restaurant,
}

Rules:

enum is only the identity, not the whole architecture
avoid using this enum directly all over the app
mode behavior must come from definitions/capabilities, not enum switch logic spread everywhere
2. BusinessModeDefinition contract

Create a contract that defines what each business mode contributes.

Suggested shape:

mode
id
displayName
capabilities
navItems
buildRoutes()
pricingEngine
receiptComposer
seedDataStrategy

Example intent:

abstract class BusinessModeDefinition {
  BusinessMode get mode;
  String get id;
  String get displayName;

  List<FeatureCapability> get capabilities;
  List<AppNavItem> get navItems;
  List<RouteBase> buildRoutes();

  OrderPricingEngine get pricingEngine;
  ReceiptComposer get receiptComposer;
  SeedDataStrategy get seedDataStrategy;
}

Rules:

keep this clean and practical
no plugin-heavy over-engineering
this contract is the main extension point for future modes
